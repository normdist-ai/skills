# VOC 实用工具与方法

本文档提供 VOC 分析过程中使用的实用工具和方法。

---

## 工具 1: 5WHY 根因分析法

用于深挖客户需求背后的真实原因。

### 模板

```
问题描述: _____________________

1. 为什么会出现这个问题？
   答: _____________________
   
2. 为什么会出现这个原因？
   答: _____________________
   
3. 为什么会出现这个更深层次的原因？
   答: _____________________
   
4. 为什么会出现这个根本原因的前置原因？
   答: _____________________
   
5. 根本原因是什么？
   答: _____________________

改进措施: _____________________
```

### 示例

```
问题描述: 用户抱怨导出功能太慢

1. 为什么导出慢？
   答: 导出时需要查询大量数据
   
2. 为什么需要查询大量数据？
   答: 用户经常导出全量数据
   
3. 为什么用户要导出全量数据？
   答: 用户不知道可以筛选后再导出
   
4. 为什么用户不知道可以筛选？
   答: 筛选功能不明显，用户没注意到
   
5. 根本原因是什么？
   答: UI 设计不合理，筛选功能可见性差

改进措施: 优化 UI 设计，提高筛选功能的可见性；添加导出引导提示
```

## 工具 2: 用户故事拆分法

将大需求拆分为可实施的小需求。

### 拆分维度

| 维度 | 示例 |
|------|------|
| **按流程** | 创建 → 编辑 → 删除 |
| **按角色** | 管理员 → 普通用户 → 访客 |
| **按数据** | 单条 → 批量 → 全量 |
| **按界面** | 列表页 → 详情页 → 编辑页 |
| **按操作** | 查看 → 搜索 → 排序 → 过滤 |
| **按性能** | 基本功能 → 性能优化 → 高级功能 |

### 示例

```
原始需求: 用户需要管理订单

拆分后:
1. 作为用户，我想要查看订单列表，以便了解我的所有订单
2. 作为用户，我想要搜索订单，以便快速找到特定订单
3. 作为用户，我想要查看订单详情，以便了解订单的具体信息
4. 作为用户，我想要取消订单，以便处理不需要的订单
5. 作为管理员，我想要导出订单，以便进行数据分析
```

## 工具 3: 需求优先级评分法

使用量化方法评估需求优先级。

### 评分公式

```
优先级分数 = (频率 × 0.3) + (严重性 × 0.3) + (商业价值 × 0.25) + (可行性 × 0.15)

其中:
- 频率: 1-5 分 (5=频繁提及)
- 严重性: 1-5 分 (5=严重影响)
- 商业价值: 1-5 分 (5=高价值)
- 可行性: 1-5 分 (5=容易实现)
```

### 评分标准

| 分数范围 | 优先级 | 行动建议 |
|---------|--------|---------|
| 4.5 - 5.0 | 关键 | 立即实施 |
| 3.5 - 4.4 | 高 | 本迭代实施 |
| 2.5 - 3.4 | 中 | 下迭代实施 |
| 1.0 - 2.4 | 低 | 放入待办列表 |

### 示例

```
需求: 数据导出功能

评分:
- 频率: 5 分 (多个用户频繁提及)
- 严重性: 4 分 (影响用户工作效率)
- 商业价值: 5 分 (提升用户满意度和留存率)
- 可行性: 5 分 (技术实现简单)

优先级分数 = (5 × 0.3) + (4 × 0.3) + (5 × 0.25) + (5 × 0.15)
           = 1.5 + 1.2 + 1.25 + 0.75
           = 4.7

结论: 关键优先级，立即实施
```

## 工具 4: 需求冲突解决矩阵

当多个需求冲突时，使用此矩阵决策。

### 决策原则

| 冲突类型 | 决策原则 | 示例 |
|---------|---------|------|
| **功能冲突** | 选择价值更高的 | 功能 A 价值 8 分，功能 B 价值 6 分 → 选择 A |
| **资源冲突** | 选择紧急度更高的 | 需求 A 紧急度高，需求 B 紧急度低 → 选择 A |
| **时间冲突** | 选择依赖更少的 | 需求 A 无依赖，需求 B 有依赖 → 选择 A |
| **用户冲突** | 选择影响用户更多的 | 需求 A 影响 80% 用户，需求 B 影响 20% 用户 → 选择 A |

### 示例

```
冲突场景: 
- 需求 A: 数据导出功能（价值 8 分，紧急度 7 分，无依赖，影响 60% 用户）
- 需求 B: 数据可视化功能（价值 7 分，紧急度 5 分，依赖 A，影响 40% 用户）

决策分析:
1. 价值: A (8) > B (7)
2. 紧急度: A (7) > B (5)
3. 依赖: A 无依赖，B 依赖 A
4. 用户影响: A (60%) > B (40%)

结论: 优先实施需求 A，然后实施需求 B
```

## 工具 5: Kano 模型问卷法

用于确定需求的 Kano 分类。

### 问卷设计

对于每个功能，询问用户两个问题：

1. **功能性问题**: 如果产品有这个功能，您感觉如何？
   - 我喜欢
   - 我期望如此
   - 我无所谓
   - 我可以忍受
   - 我不喜欢

2. **反功能问题**: 如果产品没有这个功能，您感觉如何？
   - 我喜欢
   - 我期望如此
   - 我无所谓
   - 我可以忍受
   - 我不喜欢

### 分类矩阵

| 功能性 \ 反功能性 | 喜欢 | 期望如此 | 无所谓 | 可以忍受 | 不喜欢 |
|------------------|------|---------|--------|---------|--------|
| **喜欢** | Q | A | A | A | O |
| **期望如此** | R | I | I | I | M |
| **无所谓** | R | I | I | I | M |
| **可以忍受** | R | I | I | I | M |
| **不喜欢** | R | R | R | R | Q |

**图例**:
- M (Must-be): 基本型需求
- O (One-dimensional): 期望型需求
- A (Attractive): 兴奋型需求
- I (Indifferent): 无差异需求
- R (Reverse): 反向需求
- Q (Questionable): 可疑结果（需要重新调查）

## 工具 6: 客户旅程地图模板

可视化客户体验，识别触点和痛点。

### 模板结构

```
阶段: 认知 → 考虑 → 购买 → 使用 → 忠诚

每个阶段记录:
- 客户行为: 客户在做什么？
- 触点: 客户通过什么渠道接触产品？
- 情感: 客户感觉如何？
- 痛点: 客户遇到什么问题？
- 机会: 我们可以如何改进？
```

## 工具 7: 情感分析词典

快速识别客户情感的关键词。

### 正面情感关键词

| 强度 | 关键词 |
|------|--------|
| 高 | 喜欢, 爱, 棒, 完美, 出色, 优秀, 太好了 |
| 中 | 满意, 不错, 还可以, 挺好, 方便, 有用 |
| 低 | 还行, 可以接受, 基本满足, 一般 |

### 负面情感关键词

| 强度 | 关键词 |
|------|--------|
| 高 | 讨厌, 恨, 糟糕, 垃圾, 失望, 气死, 无法忍受 |
| 中 | 不满, 困惑, 麻烦, 不方便, 失望, 担心 |
| 低 | 略有不满, 小问题, 稍微, 有点 |

### 紧急程度关键词

| 紧急程度 | 关键词 |
|---------|--------|
| 立即 | 紧急, 马上, 立刻, 现在, 急需, 崩溃 |
| 尽快 | 希望, 期待, 尽快, 早日, 下版本 |
| 正常 | 建议, 可以, 如果, 方便的话 |
| 低优先级 | 有空, 以后, 未来, 长期 |

## 工具 8: 需求验证检查清单

确保需求质量的检查清单。

### 完整性检查

- [ ] 需求描述是否清晰明确？
- [ ] 是否包含用户故事？
- [ ] 是否定义了验收标准？
- [ ] 是否识别了依赖关系？
- [ ] 是否评估了风险？

### 一致性检查

- [ ] 需求之间是否有冲突？
- [ ] 是否与现有功能冲突？
- [ ] 是否与业务规则一致？
- [ ] 是否与用户原话一致？

### 可测试性检查

- [ ] 验收标准是否可测试？
- [ ] 是否定义了测试用例？
- [ ] 是否有明确的预期结果？
- [ ] 是否可以自动化测试？

### 可追溯性检查

- [ ] 是否链接到客户之声？
- [ ] 是否链接到分析结果？
- [ ] 是否有唯一 ID？
- [ ] 是否更新了追溯矩阵？
