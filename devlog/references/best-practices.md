# DevLog 最佳实践指南

## 核心原则

1. **价值导向**: 记录有价值的信息,避免流水账。✅ "优化查询性能: 2s→200ms" ❌ "做了数据库优化"
2. **可追溯性**: 关键信息关联来源。✅ "修复bug (commit: a1b2c3d, Issue #123)" ❌ "修了个bug"
3. **具体明确**: 使用具体描述和数据。✅ "工作7.5小时,完成3个任务" ❌ "工作了大半天"
4. **问题导向**: 详细记录问题和解决方案,积累知识库
5. **一致性**: 保持术语、格式、风格统一

## 内容编写规范

### 任务描述

结构: `[状态] 任务名称: 详细说明 + 关键成果`

✅ 好: `- [x] 开发用户管理模块: 实现CRUD接口,完成单元测试,覆盖率85%`
❌ 差: `- [x] 做了用户模块`

### 问题描述

完整结构:
```markdown
### 问题: [简短标题]
- **发生时间**: HH:MM
- **问题详情**: 现象、影响、发现方式
- **根本原因**: 技术层面分析
- **解决方案**: 具体步骤
- **耗时**: X小时
- **状态**: ✅/🔄/⏸️
- **相关资源**: commit/Issue/链接
```

简化版(小问题):
```markdown
### 问题: API返回500错误
- **原因**: 未处理的空指针异常
- **解决**: 添加null检查
- **耗时**: 30分钟
- **状态**: ✅ 已解决
```

### 问题分类

按严重程度: 🔴 Critical(系统不可用) / 🟡 Major(部分功能受影响) / 🟢 Minor(轻微影响)
按类型: 🐛 Bug / ⚙️ Config / 📊 Performance / 🔒 Security / 📚 Documentation

## 信息提取规则

### 从对话中提取

| 用户表达 | 提取类型 |
|---------|---------|
| "完成了XXX" | 完成事项 |
| "修复了XXX" | 完成事项+问题 |
| "遇到了XXX" | 问题追踪 |
| "花了X小时" | 时间信息 |
| "明天要XXX" | 明日计划 |

### 从Git中提取

Commit类型映射: `feat:` → 新功能, `fix:` → Bug修复, `refactor:` → 重构, `docs:` → 文档, `test:` → 测试

文件类型推断: `.py/.js/.java/.go` → 代码开发, `.md/.rst` → 文档编写, `.yml/.yaml` → 配置修改, `test_*` → 测试编写

### 信息冲突处理

1. 时间冲突: 用户明确提供 > Git时间 > 文件推断
2. 任务描述: Git commit + 对话细节合并,去重
3. 缺失信息: Git有提交但无对话 → 用commit message; 反之标注"未提交"

## 常见场景

### 多项目并行
按项目维度组织日志,在完成事项中标注项目名,在统计中分项目计算。

### 紧急故障处理
使用专用模板: 故障概述(发生时间/发现方式/影响范围/恢复时间) → 处理过程(时间线) → 根本原因 → 改进措施 → 后续跟进

### 学习和调研
记录: 学习主题/时间/内容/关键收获/决策/参考资料

## 效率指标

- 任务完成率 = 完成任务数 / 计划任务数 × 100%
- 平均任务耗时 = 总工作时长 / 完成任务数
- 代码产出效率 = 代码行数 / 工作时长 (行/小时)
- 问题解决率 = 已解决问题数 / 遇到问题总数 × 100%
